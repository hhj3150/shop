## 0. 한 줄 결론

매일 운영 동선의 큰 마찰 5개를 **5개 독립 PR**로 순차 제거한다. 위험·임팩트순으로 🔴(전역검색·대시보드) 먼저, 🟠(360 액션·송장 일괄·모바일) 뒤에. 각 PR은 `tsc 0 + 테스트 통과` 후 머지하는 단위다.

## 1. 제약·원칙

- **보존 우선**: 라이브 ERP. 기존 파생값·SSOT(`buildRosterForDate` 등) 재사용, 신규 집계 로직 금지.
- **파일 분할**: `app/admin/page.tsx`는 이미 2207줄(800줄 한도 초과). 신규 기능은 별도 컴포넌트로 추출한다.
- **법적 안전(정보통신망법)**: 단건 문자발송은 **정보성(거래·CS) 전용**. 광고성은 기존 단체문자 `(광고)` 경로만 사용.
- **불변성**: 상태 갱신은 새 객체 생성(스프레드), 직접 변이 금지.

## 2. PR 분해

### PR1 🔴 전역 검색바
- **신규**: `components/AdminGlobalSearch.tsx`
- **위치**: 관리자 헤더(역할 탭 위) 항상 표시 — 어느 탭에서든 검색.
- **대상**: 이미 로드된 `profiles`(이름·전화·주소·입금자명) + `orders`(주문번호·받는분·전화·입금자). 클라이언트 필터, 디바운스. **신규 쿼리 없음**.
- **결과 클릭**:
  - 회원(user_id 있음) → `setSelectedMember(user_id)`로 **Customer360 드로어 즉시 오픈**.
  - 게스트 주문(user_id 없음) → 기존 `focusOrderInManageTab(order_no)` 폴백.
- **테스트**: 검색 매칭 순수함수(이름/전화/주문번호/입금자 부분일치, 정규화) 단위테스트.

### PR2 🔴 '오늘 할 일' 대시보드
- **신규**: `components/AdminTodayBoard.tsx`. 종합 관리 탭 **최상단**(기존 개요 Stat 위).
- **카드(전부 기존 파생값 props 주입)**: 입금확인→배송준비 대기(`depositPendingCount`) · 입금대기 PayAction 미등록 · 오늘 발송 대상(today roster) · 해지·환불 처리 대기 · 이상감지 건수(`anomalies`).
- **동작**: 각 카드 클릭 → 해당 탭 점프(발송 대상→배송, 환불 대기→환불·교환 등). 헤더 `입금확인 N건` 배지에 **클릭 이동** 부여(현재 무동작 회귀).
- **테스트**: 보드는 표시 컴포넌트라 렌더 스냅샷 수준. 집계는 기존 테스트가 커버.

### PR3 🟠 Customer360 문자발송 + CS메모
- **DB**: `supabase/migration-admin-memo.sql` — `profiles.admin_memo text` 추가(감사 권고). `is_admin()` RLS update 경로.
- **CS메모**: 360 드로어에서 편집·저장. `lib/customer-360`에 memo 필드 노출.
- **문자발송**: 드로어 "문자 보내기" 박스 → 기존 `POST /api/broadcast { isAd:false, recipients:[phone], message }` 재사용.
  - **법적**: isAd:false = 정보성(야간차단·(광고)·동의필터 없음). UI에 **"정보성(주문·CS)만"** 라벨 고정. 광고성 발송은 단체문자 패널로 유도.
  - sms_log에 기록 → **복기 타임라인에 자동 표시**(추가 작업 불필요).
- **테스트**: 메모 저장/로드 경로 + 발송 payload(isAd:false 강제) 검증.

### PR4 🟠 송장 일괄 붙여넣기
- **수정**: `components/DispatchPanel.tsx`
- 엑셀 붙여넣기(`주문번호<TAB>송장번호` 행들) → 파싱 후 **주문번호 매칭**으로 큐 각 행 `tracking` state 자동 채움 → 운영자 검토 후 기존 일괄 발송. 매칭 실패 행은 경고.
- **테스트**: 붙여넣기 파서(TSV/CSV, 공백·헤더 허용, 매칭/미매칭 분리) 순수함수 단위테스트.

### PR5 🟠 모바일 카드뷰
- **수정**: 주문 관리 표(주문·입금 탭).
- 좁은 화면(<sm)에서 가로스크롤 표 대신 **세로 카드** 렌더. 우선순위: 매일 보는 주문 관리 표 1개. 회원표 등은 후속.
- **테스트**: 반응형 표시이므로 시각 검증(스크린샷) 중심.

## 3. 순서·의존성

PR1 → PR2 → PR3 → PR4 → PR5. 상호 독립이나 위험·임팩트순으로 진행. PR3만 prod SQL 수동 적용 필요(`migration-admin-memo.sql`), 미적용 시 메모/발송은 best-effort no-op로 무해하게 설계.
