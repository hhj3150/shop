# 로젠 엑셀 송장 일괄 기재 — 설계

작성일: 2026-06-12

## 1. 배경·목표

로젠택배 Open API 연동은 secretKey 미발급·고정IP 미정으로 블로킹 상태다. 그 우회로로,
로젠 웹에서 내려받는 **`주문실적조회 (날짜).xlsx`** 를 관리자 배송 화면에 업로드해
주문별 **운송장번호를 일괄 기재**하고 기존 '선택 발송' 플로우(`decideShipOut`)로 연결한다.

기존 #62 "송장 일괄 붙여넣기"(`lib/tracking-paste.ts`)는 **주문번호 → 송장번호** 정확매칭을
전제한다. 그러나 로젠 엑셀은 주문번호 칸이 비어 있어(아래 §2) 그대로는 쓸 수 없다. 본 설계는
**수하인명 + 휴대폰 앞7자리** 매칭 + **운영자 수동 승인** 경로를 추가한다.

## 2. 입력 파일 실측 구조 (2026-06-08 샘플, 21건)

- 시트명 `주문실적조회`. 제목행(row2), **헤더 2줄 병합(row4·row5)**, 데이터 **row6~**, 마지막은 합계행(운송장 빈칸).
- 핵심 컬럼(1-base):
  - **col8 `주문번호` — 21행 전부 빈칸** (현재 다운로드엔 우리 주문번호가 안 들어옴)
  - **col9 `운송장번호`** — 예 `445-3834-1186` (우리가 채울 값)
  - **col13 `수하인명`** — 대부분 평문. 변형: `이일석대표`(직함 접미), `박미영(문성권)`(괄호)
  - **col17 `휴대폰`** — 앞 7자리만 노출, 뒤 4자리 마스킹 `010-7663-****`
  - col14 우편번호(앞2자리만)·col15 주소(건물번호 마스킹) — 매칭 보조로도 신뢰 낮음 → 미사용
- 매칭 가용 키: **수하인명(col13)** + **휴대폰 앞7자리(col17)** 뿐.

## 3. 매칭 대상(우리 데이터)

후보 집합은 **`allRows`(필터 적용 전 발송 가능 전체)** 로 한다 — 기존 #62 붙여넣기와 동일.
화면 필터(날짜·검색·유형·상태)로 가려진 `queue`를 쓰면 가려진 주문이 거짓 `unmatched`로 빠지므로
`allRows`를 쓴다. 단 그 결과 `unmatched`는 "필터로 가려진 것"일 수 있으니 미리보기에 안내한다.

- 실제 행은 중첩 구조다: `DispatchRow.o`(=`DispatchOrder`)에 주문 필드가 있다.
  매칭 순수함수에는 **`allRows.map(r => r.o)`** 의 부분집합 `{id, order_no, ship_name, ship_phone, tracking_no}`
  를 넘긴다(어댑터는 컴포넌트 책임).

## 4. 매칭 규칙

행 단위로 다음 순서:

1. **주문번호 정확매칭(미래 대비):** 로젠 col8(주문번호)이 채워져 있고 우리 `order_no` 와
   일치하면 확정 매칭. (로젠 접수 시 SY주문번호를 넣게 되면 자동으로 이 경로를 탄다 — 무비용 업그레이드)
2. **이름+휴대폰 매칭(현행 기본):**
   - 휴대폰 앞7자리 일치를 **필수 키**로 한다. 양쪽 휴대폰을 **신규 중립 유틸 `lib/phone.ts`의
     `normalizePhone`**(`+82`→`0` 변환, 비숫자 제거)으로 `010XXXXXXXX` 표준형으로 만든 뒤 **앞 7자리** 비교.
     - ⚠ `lib/payaction.ts`는 서버 전용(브라우저 노출 금지)이라 거기서 import 금지. 동일 로직을
       `lib/phone.ts`로 **추출**하고 `payaction.ts`·`logen-match.ts`(클라 경로)가 함께 참조한다.
       (`lib/duplicates.ts`의 중복 `normalizePhone`도 같은 집으로 통합 — 선택)
     정규화 결과가 7자리 미만이면 그 키는 무효 → 해당 행은 휴대폰 매칭 불가(이름만으로는 매칭 안 함 → unmatched).
   - 이름은 **양쪽 정규화 후 정확비교**(부분일치 아님)로 확신도를 정한다. 정규화 = 공백 제거 +
     괄호 `(...)` 제거 + **알려진 직함 접미 제거**(`대표`,`사장`,`님`,`원장`,`점장` 등 화이트리스트).
     화이트리스트에 없는 장식(`실장`·`상회` 등)이 남아 불일치하면 **`review`로 강등(unmatched 아님)**.
   - **확신도:**
     - `high` — 휴대폰7 일치 **그리고** 이름(정규화) 정확일치 → 미리보기 기본 체크 ✅
     - `review` — 휴대폰7 일치 **그러나** 이름 불일치(선물·괄호 수령인·미등록 직함 등) → 기본 체크 해제, 운영자 확인
3. **모호(ambiguous):**
   - (a) 한 로젠 행에 휴대폰7로 2건 이상 우리 주문이 걸림 → 후보 중 운영자 선택.
   - (b) **역방향**: 서로 다른 두 로젠 행이 **같은 우리 주문**을 매칭(분할발송·중복 export 라인·휴대폰7 공유)
     → 그 주문을 점유한 행들을 ambiguous로 묶어 운영자가 1건만 고르게 한다(말없는 덮어쓰기 금지).
4. **미일치(unmatched):** 로젠 행이 어떤 우리 주문과도 안 걸림 → 표시만, 무시 가능.
5. **이미 채움(already-filled):** 매칭된 우리 주문에 `tracking_no`가 이미 있으면 **자동삽입 금지** →
   미리보기에 "이미 송장 있음" 별도 상태로 표시(같은 파일 재업로드·재실행 시 올바른 송장 덮어쓰기 방지).

> 자동삽입은 `high` 이면서 기존 송장이 없는 행만. `review`·`ambiguous`·`unmatched`·`already-filled`
> 은 운영자 명시 승인 없이는 송장칸을 안 채운다.

## 5. 컴포넌트 설계

### 5.1 `lib/logen-excel.ts` (신규·순수)
- 입력: 파싱된 시트의 2차원 셀 배열(문자열). (xlsx→배열 변환은 호출부 책임 — §5.3)
- **헤더 텍스트로 컬럼 인덱스 탐지**(`운송장번호`/`수하인`/`휴대폰`/`주문번호`) → 컬럼 드리프트 견고.
  - 헤더는 **2줄 병합(row4·row5)**. SheetJS는 병합셀 값을 좌상단 원점 셀에만 채우므로,
    **헤더 후보 밴드(상위 몇 행)를 위→아래로 스캔**해 각 라벨이 처음 나타나는 (행,열)을 잡고
    그 **열 인덱스**를 컬럼 매핑으로 쓴다(휴대폰 라벨이 row5에 있어도 열은 동일).
- 출력: `LogenRow[] = { tracking, recipientName, phone7, orderNo }`.
  - `tracking`: col9 **숫자만 추출(하이픈 제거)** — 로젠 송장은 11자리 숫자, `couriers.ts`의 logen
    trace URL(`/trace/${no}`)이 숫자형을 기대하므로 저장도 숫자형으로 통일. 빈 행/합계행은 스킵.
  - `phone7`: col17 → `normalizePhone` → 앞 7자리(무효면 빈문자).
  - `orderNo`: col8(보통 빈문자).
- 견고성: 운송장번호 없는 행 스킵(합계행·빈행), 헤더 못 찾으면 빈 배열 + 사유.

### 5.2 `lib/logen-match.ts` (신규·순수)
- 입력: `LogenRow[]`, 후보 주문 `{id, order_no, ship_name, ship_phone, tracking_no}[]`(=`allRows.map(r=>r.o)`의 부분집합).
- 출력:
  ```
  { matched: { rowIdx, orderId, tracking, confidence: 'high'|'review' }[],
    alreadyFilled: { rowIdx, orderId, tracking }[],   // 매칭됐으나 주문에 송장이 이미 있음
    ambiguous: { rowIdx, tracking, candidateOrderIds }[],  // 행→다건 또는 주문 중복점유
    unmatched: { rowIdx, tracking, recipientName, phone7 }[] }
  ```
- 처리 순서: ①행별 후보 산출(주문번호 정확 → 휴대폰7+이름) ②행↔주문 다대일/일대다 충돌 검출해
  ambiguous 로 격리 ③살아남은 매칭 중 주문에 기존 `tracking_no` 있으면 alreadyFilled 로 분리.
- 이름 정규화·휴대폰7 추출 헬퍼는 순수 함수. 휴대폰 정규화는 신규 `lib/phone.ts`의 `normalizePhone` 사용
  (서버전용 `payaction.ts` 비의존). 직함 화이트리스트는 상수.

### 5.3 `components/DispatchPanel.tsx` (수정)
- 기존 접이식 '붙여넣기 박스' 옆/아래에 **로젠 엑셀 업로드** 영역 추가.
- 파일 선택(`<input type=file accept=".xlsx">`) → **SheetJS(`xlsx`)로 클라이언트 파싱** →
  셀 2차원 배열 → `parseLogenSheet` → `matchLogen` → **미리보기 테이블** 렌더.
- 미리보기: 로젠행(수하인·휴대폰·송장) → 매칭된 우리 주문(주문번호·이름) + 확신도 배지 + 체크박스.
  `high`(기존 송장 없음) 기본 체크, `review`/`ambiguous`/`alreadyFilled` 해제. `ambiguous` 는 후보 중 선택 드롭다운.
- **택배사 가드:** 로젠 파일이므로 매칭 적용 시 상단 `courier`를 **`logen`으로 자동 설정**(다른 값이면 경고)
  → 송장 조회 링크·발송 문자 택배사 오표기 방지.
- "선택분 송장 채우기" → 승인된 행만 기존 `tracking[]` state 를 **불변 갱신**(스프레드) →
  매칭된 주문 행 자동선택 → 이후는 기존 '선택 발송'(`decideShipOut`) 그대로.
- 요약 표시: 채움 N / 검토 M / 이미채움 F / 미일치 K. `unmatched`엔 "필터로 가려졌을 수 있음" 주석.

### 5.4 의존성
- **`xlsx`(SheetJS) 추가** — 관리자 전용 화면 한정 사용. `DispatchPanel`은 클라이언트 컴포넌트이므로
  파일 선택 핸들러에서 **동적 import(`await import('xlsx')`)** 하여 메인 번들·SSR 영향 차단.

## 6. 비범위(Out of scope)

- 로젠 Open API 직접 연동(별도 트랙, secretKey 대기).
- 로젠 접수 시 우리 주문번호를 col8에 넣는 작업(운영 프로세스 — 되면 §4-1로 자동 수혜).
- 우편번호/주소 기반 매칭(마스킹으로 신뢰 낮음).
- 발송 자체 로직 변경 — `decideShipOut`·문자 발송은 불변.

## 7. 테스트

- `lib/logen-excel.test.ts`: 실측 2줄 병합헤더 파싱, 합계행 스킵, 컬럼 드리프트(헤더 탐지),
  하이픈 송장 숫자화, 휴대폰7 추출, col8 빈값.
- `lib/logen-match.test.ts`: high/review/ambiguous/unmatched/alreadyFilled 각 케이스,
  이름 정규화(`이일석대표`→`이일석`, `박미영(문성권)`→`박미영`, 미등록직함→review),
  **`+82` 휴대폰 정규화 매칭**, 휴대폰7 무효(짧음)→unmatched, 두 로젠행이 한 주문 점유→ambiguous,
  col8 정확매칭 우선.
- 순수 함수 위주 — 컴포넌트는 상태 불변 갱신만, 로직은 lib에.

## 8. 리스크·완화

- **오매칭 → 택배 오발송:** `high` 외 자동삽입 금지 + 운영자 미리보기 승인으로 차단.
- **동명이인+동일 휴대폰7:** ambiguous 로 빠져 수동 선택.
- **컬럼/헤더 변경:** 헤더 텍스트 탐지로 흡수, 못 찾으면 명확한 오류.
- **SheetJS 번들:** 동적 import + 관리자 라우트 한정.
